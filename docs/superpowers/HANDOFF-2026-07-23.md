## 4. Environment / architecture gotchas (learned the hard way)

- **Frozen-at-import env trap:** `config.py` freezes `JIRA_BASE_URL`/`JIRA_EMAIL`/`JIRA_TOKEN`
  at import. Editing config or `.secrets.env` needs a **backend restart** to take effect (Jira
  path). `_auth()` now reads the token live, but base URL is still frozen at import.
- **Stale process on :8000:** repeated "still broken after fix" was almost always a stale/duplicate
  backend answering :8000. Kill by **listener only**: `lsof -ti tcp:8000 -sTCP:LISTEN | xargs kill -9`
  (bare `lsof -ti :8000` also matches the browser's client connection — don't kill that).
- **Config dirs:** dev app reads `backend/instance.config.json` + `backend/.secrets.env`.
  npm package redirects to **`~/.verdikt`** via `SCRIBE_CONFIG_DIR`/`SCRIBE_CONFIG` (set by the
  launcher); evidence at `~/evidence`; streams/DB at `~/qa-dashboard`.
- **Skill is generated per-install** at onboarding (`render_skill` → `SKILL.md`). After editing
  product knowledge, hit **Rebuild skill** on the Application Profile page, or it's stale.
- **Env mode** (`environments.mode`): `static`/`local`/`deployed` = test-only (no build/deploy);
  `script` = build+deploy. A standalone Mac can't build (no `deploycli`/k8s) → keep it
  `deployed`/`static`. Set `environments.mode = "deployed"` in `~/.verdikt/instance.config.json`
  so **every** Start path is test-only.

## 5. OUTSTANDING — the real work: QA-run quality gap

**Symptoms on the Mac:** run appears stuck at "testing 5%" until refresh; report shows all
"unknown" test cases, screenshots only, no markup/assertions/score — very shallow testing.

**Desired behavior (user):** agents should review the ticket → its **comments/conversation** →
the **PR** → **product knowledge**, then produce **full test cases + PR analysis of what changed.**

### Diagnosis (evidence-based)
1. **Empty report / "unknown" / no score = the agent had nothing to analyze.** The run fetches
   the linked **PR diff** (`qa_orchestrator.py` → `qa_reconcile.fetch_ticket_pr_refs` → GitHub/
   Bitbucket API). It **fails silently** when the VCS token isn't set, the ticket has no linked
   PR/dev-info, or the repo is unreachable. Observed: a run BLOCKED with *"PR diffs were
   unretrievable (404 / 401 Bad credentials) → zero code reviewed."* With no PR + no product
   knowledge, the skill does a shallow UI poke → screenshots, no score → "unknown". **It degrades
   silently instead of saying why.**
2. **Comments/conversation are NEVER fetched** — confirmed (no `comment` access in `qa_targets`/
   `qa_runner`/`jira_client` run path). The "review comments" step doesn't exist yet.
3. **`productQA`** (critical flows / risk areas — persisted this session) only reaches the skill
   if captured at onboarding AND the skill was rebuilt. Empty → generic shallow tests.
4. **Stuck at 5% = SSE delivery bug**, not a stuck run. `qa_runner.py:134` emits `progress pct:5`
   then streams the `claude` subprocess output; events are persisted to the stream file but the
   browser's live SSE isn't flushing them until a refresh replays from disk. Look at `streams.py`
   / `EventSourceResponse` / the frontend SSE subscription.

### Prioritized plan
| P | Fix | Size | Where |
|---|---|---|---|
| P0 | **Fail loudly** when PR/VCS token missing — turn silent "unknown" into "couldn't read the PR — set VCS token / link a PR" | small | `qa_orchestrator.py`, `qa_reconcile.py`, lane blocker |
| P0 | **SSE progress delivery** (stuck at 5%) | medium | `streams.py`, `server.py` SSE route, frontend `subscribeSSE` |
| P1 | **Fetch ticket comments** + richer PR analysis into the run scope (the desired flow) | medium-large | `jira_client.py` (add comments fetch), `qa_targets.py` (merge into scope), skill template `backend/templates/qa-evidence.skill.base.md` |
| P1 | Enforce `productQA` + auto-rebuild skill on the Mac | small | onboarding/profile |
